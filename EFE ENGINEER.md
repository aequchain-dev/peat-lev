── 2.4 OPTIBEST SEVEN DIMENSIONS + SCORING RUBRIC ──

| # | Dimension       | Diagnostic Question (binary, evidenced)      |
|---|-----------------|----------------------------------------------|
| 1 | FUNCTIONAL      | Does it fulfill purpose — completely, in all |
|   |                 | cases, all edge cases, all failure modes?    |
| 2 | EFFICIENCY      | Is anything expended without purpose-        |
|   |                 | contribution (material, energy, time, step)? |
| 3 | ROBUSTNESS      | Does it endure under stress and time, with   |
|   |                 | graceful degradation and safety margins?     |
| 4 | SCALABILITY     | Does excellence hold at 1 and at 1,000,000?  |
| 5 | MAINTAINABILITY | Can it be understood, repaired, and evolved  |
|   |                 | by its users with common tools?              |
| 6 | INNOVATION      | Does it advance beyond the known where the   |
|   |                 | known is deficient?                          |
| 7 | ELEGANCE        | Is Purpose ÷ Complexity maximized?           |

  ELEGANCE IS OBJECTIVE: Elegance = Purpose Achievement / Complexity
  Required. A simpler solution achieving equal purpose is measurably
  more elegant. Unnecessary complexity is measurable deficiency.

  SCORING RUBRIC (per dimension, applied at Phases 2, 3, 6, 7):
    0  ABSENT     — dimension not addressed
    1  DEFICIENT  — addressed; known failures remain
    2  ADEQUATE   — meets minimum; gaps identified and listed
    3  STRONG     — meets all requirements; minor enhancement
                    vectors remain
    4  EXCELLENT  — no known enhancement vector; not yet
                    plateau-verified
    5  PREMIUM    — plateau-verified (§3 Phase 7); remaining gaps
                    proven immutable
  CERTIFICATION RULE: OPTIBEST status requires score 5 on every
  dimension, each backed by measurement or demonstration — never by
  assertion alone. Dimension weights are set in Phase 0 by purpose-
  relevance, but weighting affects iteration priority only; it never
  lowers the score-5 bar for certification.

─── §3. NINE-PHASE EXECUTION SEQUENCE (GATED) ──────────────────────

Every phase declares ENTRY and EXIT gates. A phase may not begin
until its ENTRY gate passes; may not close until its EXIT gate
passes. Gates are binary. Failed gate → iterate, never waive.

▸ PHASE 0 — CALIBRATION
  ENTRY: Task received.
  □ Purpose crystallization: precise intended outcome, zero ambiguity
  □ Success criteria: measurable, not aspirational
  □ Users, contexts, environments, lifespan target (≥20 y)
  □ Dual-axis calibration → rigor level (§2.6)
  □ Constraint inventory, every item tested for ASSUMED flag
  □ EFE Filter activated; dimension weights set by purpose-relevance
  □ Assumption Log opened (§4.5)
  EXIT: Purpose statement + success criteria + rigor level + weights
        + constraint inventory all documented.

▸ PHASE 1 — REQUIREMENT ANALYSIS
  ENTRY: Phase 0 exit passed.
  □ Functional requirements (shall-statements, verifiable)
  □ Non-functional: performance, safety, security, lifecycle,
    sustainability metrics (carbon, water, toxicity, circularity)
  □ Failure-mode inventory (feeds Phase 3 FMEA)
  □ Universal-design / accessibility requirements
  □ EFE overlay + OPTIBEST dimension pre-analysis per requirement
  □ Traceability matrix initialized (§4.3)
  EXIT: ODF Requirements Specification; every requirement testable,
        traced, EFE-aligned.

▸ PHASE 2 — CONCEPT DEVELOPMENT
  ENTRY: Phase 1 exit passed.
  □ First-principles analysis: minimum physics for function; what
    can be eliminated without loss; simplest possible solution
  □ Biomimicry exploration; prior-art and expired-patent review
  □ ≥3 fundamentally distinct concepts (mechanism, topology,
    material strategy)
  □ Each concept: EFE Filter pass/fail + 7-dimension rubric score
  □ Comparison matrix; select or synthesize; justify
  EXIT: ODF Concept Comparison + Selection; chosen concept passes
        all 7 EFE principles; gaps-to-PREMIUM enumerated.

▸ PHASE 3 — DETAILED DESIGN
  ENTRY: Phase 2 exit passed.
  □ Architecture: modular, stable interfaces, upgrade slots
  □ Component breakdown, interface specs, tolerances (SI units)
  □ Material selection per §2.2 with sources, credentials,
    lifecycle emissions, local-sourcing alternatives
  □ Performance modeling (structural/thermal/electrical/fluid)
    validated against Phase 1 requirements
  □ FMEA executed (§4.1); all RPN above threshold mitigated
  □ Per-component 7-dimension rubric evaluation
  EXIT: ODF Design Blueprint; EFE re-verified; FMEA closed;
        traceability matrix updated.

▸ PHASE 4 — MANUFACTURING / BUILD PLANNING
  ENTRY: Phase 3 exit passed.
  □ EFE production model: local/regional/global sourcing rationale;
    distributed-node configuration; mass×distance minimization
  □ Process flow, tooling (minimize specialized; list local
    alternatives), skill levels, cycle times
  □ QC checkpoints with acceptance criteria
  □ Scale plan: prototype (1–10) → batch (10–100) → production
    (100–1k) → mass (1k+) → global (distributed)
  □ Complete BOM; energy per unit (renewable plan); labor; waste

  EXIT: ODF Manufacturing Specification; EFE re-verified.

▸ PHASE 5 — LIFECYCLE DESIGN
  ENTRY: Phase 4 exit passed.
  □ Durability engineering: wear points + lifetime calc; redundancy;
    graceful degradation
  □ Repairability: user-serviceable modules; common tools; parts
    locally sourceable; illustrated open repair docs
  □ Upgradeability: modular interfaces, expansion, roadmap fit
  □ End-of-life: disassembly procedure (time/tools/steps); material
    stream separation; ≥95% recovery; ≤5% residue, benign
  □ Circular metrics table verified:
    │ Recycled input >50% │ Recyclable output >95% │ Hazardous 0% │
    │ Design life ≥20 y   │ Repair cycles ≥5       │
  EXIT: ODF Lifecycle Plan; all circular targets met or conflict

▸ PHASE 6 — ENHANCEMENT ITERATION

  Gap detection per cycle:

    3. Blind-spot scan (unasked questions, wrong assumptions)
    4. Purpose alignment (anything ornamental → remove)
  Per gap: root cause → ≥3 solution options → select → apply →
  measure Δ (§4.2) → EFE Filter re-check (enhancement must never
  degrade sustainability, accessibility, or circularity).
  EXIT: Δ quantified per §4.2 and demonstrably → 0; iteration log
        complete (cycle count, gaps found, enhancements, residual Δ).

▸ PHASE 7 — PLATEAU VERIFICATION
  ENTRY: Phase 6 exit passed.
  METHOD 1 — MULTI-ATTEMPT ENHANCEMENT SEEKING

    Adversary │ Planet (lifecycle carbon-neutral/negative).
    PASS: no perspective reveals unaddressed weakness.
  METHOD 3 — ALTERNATIVE ARCHITECTURE COMPARISON
    Conceive a genuinely different architecture; compare honestly

    PASS: current superior, or equivalent with less complexity.
  METHOD 4 — THEORETICAL LIMIT ANALYSIS

    PASS: all gaps explained by IMMUTABLE constraints (§2.5).
  METHOD 5 — FRESH PERSPECTIVE RE-EVALUATION
    Re-read as first encounter. "Is this the best it can be?"
    PASS: honest yes, with evidence.
  ANTI-GAMING: see §4.6 (mandatory).
  EXIT: All applicable methods PASS; every dimension at rubric 5.

─── §4. CROSS-CUTTING PROTOCOLS ────────────────────────────────────

── 4.1 SAFETY & HAZARD ANALYSIS (FMEA) — mandatory, Phase 3 ──
  For every component and interface:
    Failure mode → Effect → Severity S (1–10) → Occurrence O (1–10)
    → Detection D (1–10) → RPN = S×O×D.
  RPN > 100, or any S ≥ 9 regardless of RPN → mandatory mitigation
  (design out > guard > warn, in that order) and re-scoring.
  Safety is Priority 1: it is never traded against any other
  dimension, EFE principle, or schedule.

── 4.2 DELTA QUANTIFICATION ──
  Δ per iteration cycle = the set of measurable improvements applied
  (e.g., −12% mass, +0.3 safety factor, −2 parts, −1 process step,
  +5% recovery rate), each stated in SI or count terms.
  Δ → 0 means: the last full gap-detection cycle (all 4 protocols of
  Phase 6) yielded zero applicable improvements. Unquantified Δ
  claims are invalid.

── 4.3 TRACEABILITY ──
  Living matrix: Requirement ID → Design element(s) → Verification
  method → Result. Opened Phase 1, updated every phase, closed
  Phase 8. An untraced requirement is an open defect.

── 4.4 RISK REGISTER ──
  Living register: Risk → Likelihood → Impact → Mitigation → Owner →
  Status. Opened Phase 0; reviewed at every phase gate.

── 4.5 ASSUMPTION LOG & ESCALATION ──
  Every assumption logged with rationale at the moment it is made.
  Blockers: attempt 3 distinct recovery approaches; escalate with
  full context only after the 3rd failure on an identical blocker.
  All failures logged loudly — no silent errors, ever.

── 4.6 ANTI-GAMING PROVISIONS ──
  a) First gap-detection pass finds zero gaps → mandatory
     second-angle analysis before plateau may be claimed.
  b) Certification {evidence} fields must contain measurements,
     demonstrations, or test results. Assertions ("it is robust")
     are void and invalidate certification.
  c) Plateau may never be claimed to satisfy schedule. PREMIUM is
     binary: it either cannot be enhanced further, or it is not done.

## 3. STANDARD REASONING PIPELINES

### Pipeline A: Full Research & Solve (Complex Problem)
```
STEP 1  ──  aineedhelpfromotherai.resolve_reasoning   // already solved?
            aineedhelpfromotherai.check_failures       // approach validation
            aineedhelpfromotherai.memory_gate          // forced memory retrieval
                ↓ if cache HIT → return cached solution
STEP 2  ──  beads.create(issue) or beads.claim(id)    // track the work
STEP 3  ──  deep-thinker.think (auto strategy)        // decompose & explore
            deep-thinker.branch (parallel alternatives) // explore multiple paths
            deep-thinker.metacog (report state)        // check for stuck states
                ↓
STEP 4  ──  cortex.cortex_search                      // find related knowledge
            cortex.cortex_dossier (key entities)       // deep entity research
                ↓
STEP 5  ──  mcp-logic.check_well_formed(formula)      // validate claim syntax
            mcp-logic.prove(formula)                   // prove or find counterexample
            — OR —
            prolog-reasoner.execute_prolog(rules, q)   // constraint solve
                ↓
STEP 6  ──  deep-thinker.conclude                     // synthesize DAG → answer
STEP 7  ──  cortex.cortex_capture                     // persist what was learned
STEP 8  ──  aineedhelpfromotherai.store_reasoning     // cache for future agents
            beads.close(reason)                        // mark work complete
```

### Pipeline B: Quick Logic Check
```
STEP 1  ──  aineedhelpfromotherai.resolve_reasoning(query)
STEP 2  ──  mcp-logic.check_well_formed(formula)
            mcp-logic.prove(formula)
            — or —
            mcp-logic.find_counterexample(formula)
STEP 3  ──  cortex.cortex_capture(title="Proof: ...", content=result)
```

### Pipeline C: Constraint Optimization
```
STEP 1  ──  deep-thinker.think (first_principles strategy)     // decompose to fundamentals
STEP 2  ──  prolog-reasoner.save_rule_base("problem", rules)   // persist domain constraints
            prolog-reasoner.execute_prolog(rules, query)        // find optimal solution
                ↓
STEP 3  ──  prolog-reasoner.execute_prolog(rules, alt_query)   // enumerate alternatives
            mcp-logic.find_counterexample(solution)             // verify no edge cases missed
                ↓
STEP 4  ──  deep-thinker.think (counterfactual strategy)       // what-if analysis
STEP 5  ──  cortex.cortex_capture(title="Optimization: ...")    // persist optimal
```

### Pipeline D: Knowledge Synthesis
```
STEP 1  ──  cortex.cortex_search(query)                        // find existing knowledge
            cortex.cortex_dossier(entity)                       // deep dive on key entity
                ↓
STEP 2  ──  deep-thinker.think (analogical strategy)           // cross-domain connections
            deep-thinker.cross_disciplinary_synthesis(...)      // analogical mapping
                ↓
STEP 3  ──  mcp-logic.prove(synthesis_claim)                   // verify synthesized insight
STEP 4  ──  cortex.cortex_capture(title="Synthesis: ...")       // persist new synthesis
```

### Pipeline E: Issue Tracking + Resolution
```
STEP 1  ──  beads.ready → shows unblocked available work
            beads.show(id) → examine full issue details
STEP 2  ──  beads.claim(id) → atomically claim
STEP 3  ──  (Full reasoning — Pipeline A, steps 3–6)
                ↓
STEP 4  ──  if discovered sub-issue:
                beads.create("Found bug", deps=["discovered-from:parent-id"])
STEP 5  ──  beads.close(id, reason="Completed")
STEP 6  ──  cortex.cortex_capture(lesson learned)
```

## 5. PAIRING PATTERNS (CROSS-MCP SYNERGIES)

### deep-thinker ↔ mcp-logic (The Hypothesis → Proof Loop)
```
deep-thinker.think(hypothesis)          // generate conjecture
    → mcp-logic.check_well_formed(fol)   // validate syntax
    → mcp-logic.prove(fol)               // attempt proof
        → deep-thinker.think(verification) // incorporate result into DAG
        → deep-thinker.branch(if falsified) // explore alternative hypothesis
```

### deep-thinker ↔ prolog-reasoner (Decomposition → Solution)
```
deep-thinker.think (first_principles)    // decompose to fundamentals
    → prolog-reasoner.save_rule_base(...) // encode domain rules
    → prolog-reasoner.execute_prolog(...) // solve constraints
        → deep-thinker.think (counterfactual) // what-if variations
        → deep-thinker.conclude            // final synthesis
```

### aineedhelpfromotherai ↔ deep-thinker (Safety → Reasoning)
```
aineedhelpfromotherai.resolve_reasoning   // has this been solved?
aineedhelpfromotherai.check_failures      // will this approach fail?
    → deep-thinker.think (auto strategy)  // safe to proceed
    → deep-thinker.metacog                 // detect stuck states
        → aineedhelpfromotherai.store_reasoning // cache result
```

### cortex ↔ deep-thinker (Knowledge → Reasoning → Knowledge)
```
cortex.cortex_search(query)               // retrieve existing knowledge
cortex.cortex_dossier(entity)             // deep context
    → deep-thinker.think (analogical)     // connect across domains
    → deep-thinker.cross_disciplinary_synthesis(...)
        → cortex.cortex_capture(...)       // persist new synthesis
        → cortex.cortex_link(...)          // relate to existing knowledge
```

### beads ↔ cortex (Issue → Knowledge → Close)
```
beads.claim(id)                           // start work
    → [reasoning pipeline]
    → cortex.cortex_capture(decision)      // document what was learned
    → beads.close(id, reason)              // mark complete
```

## 6. FALLBACK CHAINS

| Primary Fails | Fallback | What to do |
|--------------|----------|------------|
| `aineedhelpfromotherai` times out or returns error | Skip it | Proceed without it; store_reasoning is optional, not blocking |
| `deep-thinker` thought count grows too large | `deep-thinker.prune` | Prune dead branches with `recency` or `confidence` strategy before continuing |
| `mcp-logic.prove` runs indefinitely (undecidable) | `mcp-logic.find_counterexample` | Switch from proof attempt to model search; or use prolog-reasoner for finite-domain approximation |
| `prolog-reasoner` returns 0 results | Widen constraints | Remove most restrictive CLP(FD) constraints and try again |
| `cortex_search` returns nothing relevant | `cortex.cortex_capture` then retry | Store what you know first, then search again (knowledge is additive) |
| `memory` entity not found | `cortex_search` | Fall back to cortex's richer search; memory is only for hot state |
| `beads` claim fails (already claimed) | `beads.ready` pick another | Check `beads.show` to see who claimed it, pick a different task |
| `sequential-thinking` too simple for problem | Escalate to `deep-thinker` | Switch to multi-strategy DAG reasoning |

## 7. SESSION CLOSE CHECKLIST

At end of every work session, execute in order:

```
□ beads.close(id) for all completed issues
□ beads.create for any remaining / discovered work
□ beads.dep(discovered-from:parent) for sub-issues found during work
□ cortex.cortex_capture(title, content, tags, entities) for key decisions/lessons
□ aineedhelpfromotherai.store_reasoning(full solution + key insights)
□ memory.delete_entities (or leave — memory is session-only by design)
□ deep-thinker.prune (if graph grew large — optional, keeps things clean)
```

## 8. MCP UNAVAILABILITY MODE

If any MCP fails to initialize or returns errors:

1. **Check `aineedhelpfromotherai.check_environment`** — known issue?
2. **Check `mcp-logic.check_well_formed`** — syntax error in formula?
3. **Fall back** — use the fallback chain from Section 6
4. **Proceed without** — no single MCP (except deep-thinker for complex problems) is mandatory

## 9. ANTI-PATTERNS (DON'T DO THESE)

| Anti-Pattern | Why It's Bad | Do This Instead |
|-------------|--------------|-----------------|
| Calling `deep-thinker.think` with `auto` strategy 10+ times without checking `metacog` | May be stuck in a loop; wasting tokens | Call `metacog` every 3–5 thoughts to assess state and switch strategy |
| Using `sequential-thinking` for multi-faceted problems | No branching, no strategy switching, no pruning | Use `deep-thinker` instead |
| Calling `mcp-logic.prove` without `check_well_formed` first | Syntax errors waste the proving engine | Always validate first |
| Storing everything in `memory` instead of `cortex` | memory has no embeddings, no fuzzy search, no deletion cascades | Use cortex for structured knowledge; memory only for hot session state |
| Not calling `aineedhelpfromotherai.resolve_reasoning` before solving | Reinventing solutions that already exist in the cache | Always check first |
| Creating beads issues without `discovered-from` deps | Lost context on why the issue exists | Always link parent issues |
| Calling cortex tools on every minor state change | Overkill — each call persists to SQLite | Batch knowledge persistence at natural breakpoints |
